# 개발 에이전트 구성 및 역할 분담

## 1. 개발 원칙

- **전문 에이전트 분업**: 각 영역 전문 에이전트가 담당 범위만 개발
- **테스트 필수**: 모든 개발 에이전트는 **자기 작업 + 테스트 코드 작성** 필수 (테스트 없는 산출물은 미완료)
- **테스터 분리**: 별도 테스터 에이전트가 통합/E2E 테스트 수행
- **Mock 우선**: 외부 API 키 발급 전까지 Mock 구현체로 개발, 인터페이스는 실제 API와 동일
- **계약 기반 핸드오프**: 에이전트 간 경계는 DB 스키마·API 인터페이스·타입 정의로 명시

---

## 2. 에이전트 목록

### 2.1 🏗️ Infra Agent (인프라/DevOps)
**담당**
- `docker-compose.yml` 전체 서비스 정의 (postgres / redis / api / web / nginx)
- Dockerfile 각 서비스별 작성
- 환경변수 관리 (`.env`, `.env.example`)
- 홈서버 배포 스크립트 (Cloudflare Tunnel 설정 포함)
- 백업/모니터링 기본 세팅

**산출물**
- `docker-compose.yml`
- `docker-compose.dev.yml` (개발용 오버라이드)
- 각 서비스 `Dockerfile`
- `.env.example`
- `README.md` (로컬 실행 가이드)

**테스트 요구**
- `docker compose up` 한 번에 모든 서비스 기동 성공
- `docker compose ps` 전부 healthy 확인
- PostgreSQL/Redis 각각 외부 연결 스모크 테스트 스크립트

**선행**: 없음 (가장 먼저)

---

### 2.2 🗄️ Database Agent (DB 스키마·마이그레이션)
**담당**
- PostgreSQL 스키마 설계 (키워드, 상품, 점수, HS코드, 히스토리, 사용자 피드백)
- Alembic 마이그레이션 (처음부터)
- Seed 데이터 (카테고리, 쿠팡 수수료 테이블, HS코드 초기본)

**산출물**
- `alembic/` 디렉토리 + 마이그레이션 파일
- `backend/models/` SQLAlchemy 모델
- `scripts/seed.py`
- 스키마 ERD 문서 (`docs/schema.md`)

**테스트 요구**
- 마이그레이션 up/down 왕복 성공 테스트
- 각 모델의 CRUD 테스트 (pytest + test DB)
- Seed 스크립트 멱등성 테스트

**선행**: Infra Agent (postgres 컨테이너 필요)

---

### 2.3 🔌 Data Collection Agent (외부 API 어댑터)
**담당**
- 외부 API 클라이언트 추상화 인터페이스 (프로토콜)
- Mock 구현체 (샘플 JSON 고정 응답, 실제 API 응답 포맷 모사)
- 실제 구현체 뼈대 (나중에 키 발급 시 채움)
- 응답 DTO (Pydantic 모델)
- 캐시 전략 (쿠팡 파트너스 1시간 10회 대응 — 24h PostgreSQL 캐시)
- Rate limiter

**대상 API** (각각 Mock + 실제 인터페이스)
- `NaverSearchAdClient` (keywordstool)
- `NaverDataLabClient`
- `NaverShoppingClient`
- `NaverBlogCafeClient`
- `CoupangPartnersClient`
- `CustomsClient` (관세청 품목별 국가별 수출입)
- `HsCodeClient` (관세청 HS부호)
- `ExchangeRateClient` (한국수출입은행)
- `YouTubeDataClient`
- `GoogleTrendsClient` (pytrends)

**산출물**
- `backend/clients/` 각 클라이언트
- `backend/clients/mocks/` 샘플 응답 JSON
- `backend/clients/base.py` 공통 인터페이스
- `backend/cache/` 캐시 레이어

**테스트 요구**
- 각 Mock 클라이언트 응답 스키마 검증 (Pydantic)
- 캐시 hit/miss 테스트
- Rate limiter 테스트 (10req/hour 초과 시 대기/에러)
- 실제 구현체는 "인터페이스 호환성만" 테스트 (키 없이 실 호출 X)

**선행**: Database Agent (캐시 테이블 필요)

---

### 2.4 🧮 Scoring Engine Agent (점수 엔진)
**담당**
- 기회 점수 계산 (100점: 수요 25 / 성장 20 / 경쟁공백 20 / 수입실체 20 / 선행 10 / 안정성 5)
- 상품 점수 계산 (기회 40 / 수익 35 / 리스크 15 / 안정 10)
- 양 채널 수익 계산 (스마트스토어 5.5% / 쿠팡 카테고리별)
- HS코드 매핑 로직 (카테고리·키워드 → HS코드)
- 자동 제외 필터 (인증 필요/계절성/레드오션 등)

**산출물**
- `backend/scoring/opportunity.py`
- `backend/scoring/product.py`
- `backend/scoring/revenue.py` (채널별 수익 계산)
- `backend/scoring/filters.py`
- `backend/scoring/hs_mapping.py`

**테스트 요구**
- 각 점수 공식 단위 테스트 (경계값: 0점, 100점, 중간)
- 수익 계산 회귀 테스트 (알려진 값 → 기대 순이익)
- 필터 테스트 (입력 케이스별 통과/차단)
- 속성 기반 테스트 권장 (Hypothesis)

**선행**: Data Collection Agent (DTO 타입 필요)

---

### 2.5 🔗 Backend API Agent (FastAPI)
**담당**
- FastAPI 엔드포인트 설계
- 라우터·의존성·인증 (나중엔 간단한 토큰 인증)
- 비즈니스 서비스 레이어 (수집/점수/조회 오케스트레이션)
- 백그라운드 작업 큐 (키워드 수집 파이프라인)
- OpenAPI 스펙 자동 노출

**주요 엔드포인트**
- `GET /opportunities` — 오늘의 기회 키워드
- `POST /products` — 사용자 상품 입력
- `GET /products/{id}` — 점수 결과
- `GET /history` — 입력 이력
- `POST /feedback` — 실제 매출 피드백

**산출물**
- `backend/routers/`
- `backend/services/`
- `backend/main.py`
- `backend/schemas/` (요청/응답 Pydantic)

**테스트 요구**
- 각 엔드포인트 통합 테스트 (TestClient)
- 해피패스 + 검증 실패 + 404/409 케이스
- Mock 클라이언트 주입으로 외부 API 없이 테스트

**선행**: Scoring Engine Agent + Data Collection Agent

---

### 2.6 ⏰ Scheduler Agent (배치/파이프라인)
**담당**
- APScheduler 기반 배치 워커 (별도 컨테이너)
- 파이프라인 정의
  - 매일: 네이버 키워드 수집 → 점수 재계산
  - 매주: 관세청 수입량 갱신
  - 10분 간격: 쿠팡 파트너스 조회 (rate limit 내)
- 작업 실패 재시도 + 알림

**산출물**
- `scheduler/` 전용 컨테이너
- `scheduler/jobs/` 각 배치 잡
- 실행 로그 DB 기록

**테스트 요구**
- 각 잡의 단위 테스트 (Mock 클라이언트 사용)
- 재시도 로직 테스트
- 파이프라인 간 의존성 테스트 (A 완료 후 B 실행)

**선행**: Backend API Agent

---

### 2.7 🎨 Frontend Agent (Next.js)
**담당**
- Next.js (App Router, TypeScript)
- 페이지: 메인 대시보드 / 상품 입력 / 상품 상세 / 이력
- 차트 (레이더·추세 — Recharts 또는 Chart.js)
- 백엔드 API 클라이언트 (OpenAPI 기반 타입 생성)
- 상태 관리 (TanStack Query)

**산출물**
- `frontend/app/` 페이지
- `frontend/components/` 공통 컴포넌트
- `frontend/lib/api/` 백엔드 클라이언트

**테스트 요구**
- 컴포넌트 단위 테스트 (Vitest + React Testing Library)
- Mock API 응답으로 렌더링 스냅샷
- 주요 폼 검증 로직 테스트

**선행**: Backend API Agent (OpenAPI 스펙 필요)

---

### 2.8 🧪 Tester Agent (통합/E2E)
**담당**
- 통합 테스트: 여러 에이전트 산출물이 합쳐졌을 때의 시나리오
- E2E 테스트: Playwright로 실제 브라우저 플로우
- 성능 테스트: 키워드 수집 파이프라인 처리 시간
- 회귀 테스트 모음 관리
- 커버리지 리포트

**테스트 시나리오**
1. 사용자가 메인 접속 → TOP 20 키워드 보임
2. 키워드 클릭 → 1688 링크 확인
3. 상품 입력 (URL+단가+MOQ) → 점수 산출 → 2채널 비교 표시
4. 이력 저장 후 재방문 시 조회
5. 피드백 입력 후 가중치 반영 확인 (Phase 4)

**산출물**
- `tests/integration/`
- `tests/e2e/` (Playwright)
- CI 파이프라인 (`.github/workflows/test.yml` 또는 로컬 스크립트)
- 커버리지 리포트 (목표: 단위 80%, 통합 70%)

**선행**: 각 개발 에이전트 산출물

## 4. 에이전트 간 계약 (핸드오프 포인트)

| From | To | 계약물 |
|------|-----|-------|
| Infra → 전체 | 모두 | docker-compose 서비스 이름/포트/환경변수 |
| Database → Data Collection | DC | 캐시 테이블 스키마 |
| Database → Backend | Backend | 엔티티 모델 |
| Data Collection → Scoring | Scoring | 응답 DTO (Pydantic) |
| Data Collection → Backend | Backend | 클라이언트 인터페이스 |
| Scoring → Backend | Backend | 점수 함수 시그니처 |
| Backend → Frontend | Frontend | OpenAPI 스펙 (자동 생성) |
| Backend → Scheduler | Scheduler | 서비스 함수 (비즈니스 로직 재사용) |
| 전체 → Tester | Tester | 시스템 통합 상태 |

## 5. 테스트 요구사항 (공통)

### 5.1 개발 에이전트 테스트 규칙
- **pytest** (백엔드), **vitest** (프론트) 기준
- 커밋 전 전체 테스트 통과 필수
- **테스트 없는 PR/산출물은 미완료로 간주**
- 단위 테스트 커버리지 담당 모듈 **80% 이상**
- 테스트 파일 규칙
  - 백엔드: `tests/unit/<module>/test_<filename>.py`
  - 프론트: `<component>.test.tsx`

### 5.2 Tester 에이전트 규칙
- 통합 테스트는 실 DB·Redis 사용 (docker-compose의 테스트 프로파일)
- E2E는 Mock API 환경에서 실행 (실 외부 API 호출 X)
- 실패 시 에이전트별 이슈로 재할당

### 5.3 CI 게이트 (로컬)
- 린트 (ruff, eslint) 통과
- 타입 체크 (mypy, tsc) 통과
- 단위 테스트 통과 (커버리지 80%+)
- 통합 테스트 통과

## 6. 에이전트 작업 지시 템플릿

각 에이전트 호출 시 아래 형식으로 지시:

```
역할: <에이전트 이름>
담당 범위: <이 문서의 해당 섹션>
입력 계약: <선행 에이전트 산출물>
요구 산출물: <파일 경로 + 기능>
테스트 요구: <단위/통합 테스트 범위>
완료 기준: 코드 + 테스트 + 테스트 통과
금지 사항: 범위 외 변경 금지, 테스트 없는 산출물 제출 금지
```

## 7. 진행 추적

- Phase별 진행 상황은 TodoWrite로 관리

- 주간 회고: 에이전트 간 계약 불일치 사례 수집 → 문서 업데이트
